= Set up a workspace in AI Data Engine
:icons: font
:imagesdir: ../media/

[.lead]
After you set up a cluster, you can set up a workspace. Workspaces allow you to segment data on the cluster, control data access for individuals, and exclude data that should not be exposed to the AI Data Engine.

If you are administering storage, you'll use System Manager to create and manage workspaces, and add data collections to support AI workloads. 

Strategies for workspace creation vary by organization. You might create workspaces based on teams, projects, data sensitivity levels, or other criteria relevant. For example, if you work in healthcare, you might segment clinical data into a workspace but leave out data pertaining to IT, legal, or other departments.

.About this task
Workspace creation is subject to system processing limits (typically up to 15 GB per day per cluster). If you create multiple workspaces in parallel or in quick succession, each workspace may take longer to process, and you may experience significant delays. 

You can monitor the status of workspace creation from the Workspaces inventory page. For best results, avoid creating many workspaces at once if you need immediate access to these features.

Data collections and Retrieval-Augmented Generation (RAG) search capabilities become available only after the workspace creation process is fully complete. 

.Before you begin

* You need _storage administer_ privileges to create workspaces and associate data collections.
* You've determined the remote and local data sources you intend to use with the workspace and with AI Data Engine.
* You've link:workspaces/prepare-data-for-aide.html[created at last one data container] that can be used by the workspace, such as a volume or an S3 bucket.
+
NOTE: Ensure NFS is enabled on the volume but that the volume is not CIFS enabled. Workspaces only support volumes with NFS. Volumes with CIFS (SMB) are not supported with workspaces.

== Create a workspace

Create a workspace and associate data containers that contain the data you want to use with AI Data Engine.

.Steps
. In System Manager, navigate to *Data Engine > Workspaces*.
. Select *Add*.
. In the *Add Workspace* dialog, select at least one available data container to associate with the workspace.
. Configure link:prepare-data-for-aide.html[peered clusters] so that the data from those clusters can be accessed within the workspace
. If you'd like to configure user access to the workspace, you can do that now or link:assign-users-to-workspace.html[wait until after the workspace is created].
. Configure a refresh interval for how often the workspace synchronizes with the associated data containers to capture new or updated data (for example, six hours).
+
TIP: Choose an interval that balances data freshness with system performance. If you add a data container to multiple workspaces, use the most aggressive (shortest) interval. To learn more, see documentation about <<Workspace refreshes and versioning,workspaces refreshes and versioning>>.

. Select *Continue*.
. In the *Finalize workspace* dialog, enter a workspace name and description.
. Select *Add* to create the workspace.

.Result
The workspace creation process takes several minutes to hours to complete, depending on the associated dataset and its file count, file size, and other factors.

Metadata will be extracted automatically for all datasources and stored in a metadata catalog that users can use to locate the files they need for their projects. After you assign users to the workspace, they can access and interact with the workspace-affiliated data collections and resources for their AI workloads.

The new workspace will show up in the Workspaces page in `Creating` state until the process is complete and the state changes to `ready`.

== Review workspace details

After workspace creation, review the workspace details.

.Steps
. Review workspace details, including total size, and percentage of cluster capacity used.
. Select the workspace name to open the details page.
. In the Overview tab, view workspace details and associated data containers.

== Workspace refreshes and versioning

Every workspace refresh creates a new version that captures the current state of all files and objects in the workspace. Versions are immutable and traceable, supporting auditing and reproducibility.

Each version includes:
* Complete metadata of all files and objects
* References to snapshots used
* Job ID for traceability

You can list all versions, view diffs between versions, and audit changes over time.

.How Versioning Works
Every workspace refresh creates a new version.

A refresh is triggered either on schedule (per the workspace's refresh interval) or manually.
The refresh captures the current state of all files/objects in the workspace's data containers.
Each version is immutable and traceable, allowing you to audit changes over time.
.What is included in a version?

The full metadata of all files/objects in the workspace at the time of refresh.
References to the specific snapshots used for extraction.
A job ID for traceability (which job created the version).
.Why is versioning important?

Enables data lineage and reproducibility for AI/ML workloads.
Supports auditing, rollback, and change tracking.
Allows you to compare versions (diffs) to see what files/objects were added, modified, or deleted.
Versioning in Practice
Workspace versioning is automatic:

You do not need to manage versions manually; each refresh creates a new version.
The system retains previous, current, and next (in-progress) versions for each workspace.
Older versions are retained per policy for traceability and can be purged as needed.
APIs and UI:

You can list all versions of a workspace.
You can view diffs between versions to understand changes.
